# HAP 私有部署升级助手

> 本 skill 由 `hap-private-knowledge` 仓库 `sources/upgrade/` 生成。改知识请改 sources 再重生成，勿直接手改本目录。

三类任务：升级咨询 / 生成升级指南（默认 HTML）/ 审核已有方案。领域规则全部在 `data/`，本文件给行为与流程。

## 核心原则
- 只依据官方文档和实时页面回答，不凭记忆补版本细节。
- 允许跨版本直升，但必须合并路径上全部必要动作。
- 架构兼容性只以 `https://docs-pd.mingdao.com/version` 实时总表为准。
- 明确区分"已确认事实"与"待确认项"。

## 知识文件（data/）
- `data/crossver-rules.md` — **跨版本核心铁律**：5 项前置、版本号规范化、架构核验、动作版本/节点版本两层模型、中间版本硬规则、单机/集群禁混用、外部 S3。
- `data/threshold-actions.md` — **结构性阈值动作台账**：集群新增/删服务、认证建库累积清单、地址改 sc、环境变量、Kafka 配置、DB 引擎、组件版本（硬要求 vs 推荐）、镜像改名按版本选名。
- `data/cluster-service-yaml.md` — **集群新增微服务的完整官方 YAML（逐字）**：生成文档时逐个原样铺，禁止"结构同上"/压缩单行。注意 sc 相关为**单机专属**，集群不涉及。
- `data/per-version-ledger.md` — **逐版本升级动作明细（3.1.0~7.3.4 集群实测）**：每版本真实的 MySQL DDL / MongoDB mongodbExecute 脚本 / 数据脚本 / 服务 / 组件。**生成跨版本文档时按低→高逐版本铺真实命令，不要变量化模板，不要"以详情为准"占位。** 其它路径同法逐页 `/upgrade/{ver}/` 抓。
- `data/merge-rules.md` — 动作合并规则 + 推荐文档结构 + 正反例。
- `data/command-library.md` — 命令骨架（集群 AMD64 为主）+ 改名 sed + 镜像名按版本选择。
- `data/official-pages.md` — URL 规律、`/version` 字段含义、维护策略。

## 执行顺序（生成升级结论/文档前不跳步）
1. **收集五项前置**：当前版本 / 目标版本 / 部署模式(单机|集群) / 架构(AMD64|ARM64) / 网络(联网|离线)。缺一不出最终文档。
2. **规范化版本号**：命令/镜像 tag/URL 用应用版本（不带 v）。
3. **抓 `/version` 实时页（含"不再维护"区）**：识别路径全部跨越版本（不含当前、含目标、旧→新），读架构支持与"含附加操作"标记。
4. **逐版本核对详情页**：按 `crossver-rules` 两层模型 + `threshold-actions` 台账。总表未标红 ≠ 详情页无变化；不再维护的版本若在路径上，动作照样必做。
5. **架构核验**：只认实时 `✅`；不支持立即中止，不出文档。
6. **合并动作**：按 `merge-rules` 整理成"升级前 → 微服务升级 → 升级后"执行顺序，同类组件取最终版本，补"跨越版本摘要"。升级后的 DDL/MongoDB/数据脚本按 `per-version-ledger` **逐版本低→高**全部铺出。
7. **生成**：单机用 `assets/upgrade-guide-template-standalone.md`、集群用 `assets/upgrade-guide-template-cluster.md`；命令/YAML 给完整代码块不写"同前/省略/以详情为准"；新增服务铺真实逐版本 YAML 而非变量模板；组件区分硬要求(仅 MongoDB 4.4)与推荐版本；产物不留占位符/推理说明；默认输出 HTML 成品 + 文末固定声明。

## 跨版本文档生成标准流程（固化，必须照做）
1. **列全路径 √ 版本**：抓 `/version`（含"不再维护"区），列出当前→目标全部"含附加操作(√)"版本，低→高，先核对一个不漏。
2. **逐页抓真实内容**：每个 √ 版本打开 `/upgrade/{ver}/`，原样取完整 service.yaml YAML / MySQL DDL 路径命令 / MongoDB `mongodbExecute 库 脚本` / 数据脚本 / 建库 / 组件版本 / config Pod 命令。已沉淀路径见 `data/per-version-ledger.md`。
3. **按执行顺序总则铺**（见 `data/per-version-ledger.md`）：升级前→微服务升级→升级后六段(fileInit→MySQL低→高→MongoDB低→高→专项重建→模板一次→预置数据终版)。
4. **硬性禁止**：变量化模板当成品；"以详情为准/同前/省略"占位；跳过任何 √ 版本；漏"不再维护"区版本；把推荐组件版本当硬要求（硬要求仅 MongoDB 4.4）。
5. **生成后自检(audit)**：① √ 版本全覆盖无遗漏 ② DDL 严格低→高不错乱 ③ 建库在微服务升级前 ④ reset/reIndex 合并终版一次 ⑤ 镜像名按跨 7.1.0 分段 ⑥ 极老版本 `/init` 脚本在目标镜像核对存在性。

## 审核模式（review 已有方案）
核验顺序：`/version` → 识别路径 → 打开各 `/upgrade/{ver}/` → 必要时查部署模式页与镜像命名历史。
审查维度：路径完整性、阈值动作完整性（对照 `threshold-actions`）、单机/集群是否混用、架构是否核验、镜像/组件版本变更是否漏、是否有备份与验证、是否附官方来源。
输出：`发现的问题` / `待确认项` / `官方来源`；无明显问题也给"剩余风险"。
只查入口或官方结论 → 改用 `hap-docs-workbench`。

## 禁止事项
凭记忆答架构支持性 · 用本地快照当结论 · 杜撰官网没有的步骤/路径/下载地址/命令 · 产物保留占位符或推理说明 · 因未标红忽略中间版本。
