本文件是升级判断的单一事实源。下游所有"别漏中间版本/别混单机集群/别凭记忆判架构"的提醒都引用这里，不重述。

## 1. 五项前置信息（缺一不下结论、不出文档）
当前版本 · 目标版本 · 部署模式(单机|集群) · 架构(AMD64|ARM64) · 网络(联网|离线)

## 2. 版本号规范化
- **应用版本**（不带 `v`）：命令、镜像 tag、脚本参数、配置文件。
- **显示版本**（带 `v`）：标题、正文。
- `v7.1` → 应用 `7.1.0` / 显示 `v7.1.0`。命令/URL/tag 一律不带 `v`。

## 3. 架构核验（只认实时 /version）
- `✅`=支持；空白/🚫/缺失/无法确认 一律视为**不支持**。
- 目标版本不支持用户架构 → **立即中止，不生成文档**。
- 禁止凭训练记忆判断某版本是否支持某架构。

## 4. 跨版本两层模型（最易出错，务必照此）
允许跨版本直接升，但必须合并路径上全部必要动作。把路径中每个版本分两类：

- **动作版本**：满足任一即是——总表标"含附加操作"、详情页出现 MySQL DDL / MongoDB DDL / `fileInit`·`s3fileInit` / MongoDB 预置数据更新 / 新增服务配置 / 镜像命名变更 / 组件升级要求。→ 内容必须并入执行文档。
- **节点版本**：被跨过但详情页无独立动作。→ 可不写进执行步骤，**但必须出现在"跨越版本摘要"里**。

### 中间版本硬规则（唯一出处）
- "总表未标红" ≠ "详情页无变化"。`6.3.0 / 6.4.0 / 6.5.0` 这类中间版本**必须逐个打开详情页核对**上述动作。
- **"不再维护"的版本若在升级路径上，详情页动作照样必须执行**。`/version` 把老版本归到"不再维护"区，只抓"维护中"区会整段漏掉路径上的老版本动作（见下方 5.1.0 阈值动作——5.1.0 已不再维护，但跨过它必做）。
- 详情页只给"参考某文档/跳转链接"时，必须继续打开该链接提取真实命令。
- 有命令→按动作版本处理；无命令→仍写进路径摘要。**绝不静默丢弃任何路径版本。**

### 版本阈值动作（跨过该阈值即触发，源 < 阈值且目标 >= 阈值时必做）
路径上跨过的版本若有结构性动作（新增/删服务、改地址、改名、组件版本、DB 引擎、环境变量），逐条并入。
**完整台账见 [[threshold-actions]]**（5.0.0~7.3.0 已逐页实时核验）。最典型一条：跨 5.1.0 时连接地址 `127.0.0.1` → `sc`。

## 5. 单机 vs 集群（唯一出处）
- 禁止混用单机与集群命令。
- 集群模式 `doc`/`ldoc` 在 `service.yaml` 改镜像版本，**不**单独 `kubectl apply`，也不写成 `ldoc.yaml`。
- 集群升级后动作尽量合并为一次进入 config Pod 顺序执行。
- 集群无存储组件(sc)升级步骤。

## 6. 用户用外部 S3 时
所有 `fileInit` 改为 `s3fileInit`。
