把路径中多版本动作整理成一份按**执行顺序**组织的可执行文档（不按版本逐章堆叠）。
版本分层（动作版本/节点版本）见 [[crossver-rules]]，此处只讲合并与结构。

## 升级前动作合并
| 动作 | 合并方式 |
|---|---|
| 创建 MongoDB 库/用户 | 一次登录、一次性建全部需要的库/用户 |
| 文档预览/扩展服务版本变化 | 只保留最终需达到的组件版本 |
| MongoDB 预置数据更新 | 只执行路径中涉及该动作的最新版本 |
| 新增服务配置（集群） | 合并为一次 `service.yaml` 修改 |
| 镜像/脚本命名变化 | 统一放升级前，一次处理 |

## 升级后动作合并
| 动作 | 合并方式 |
|---|---|
| MySQL DDL / MongoDB DDL | 保留版本顺序，从低到高执行 |
| `fileInit` / `s3fileInit` | 同类初始化尽量只执行一次入口 |
| 进入 config Pod | 合并成一次进入后顺序执行所有命令 |

## 资源准备区
- 只列本次最终实际需要的镜像/脚本/离线资源。
- 同类组件跨多版本升级只列最终目标版本资源；但某中间版本脚本必须执行时，其版本号仍保留在命令区。

## 推荐文档结构（按执行顺序）
1. 升级结论与适用条件 → 2. 跨越版本摘要 → 3. 执行前确认 → 4. 提前准备与资源 → 5. 升级前动作 → 6. HAP 微服务升级 → 7. 升级后动作 → 8. 升级后验证

## 反例 / 正例
- ✗ 因 `6.3.0` 未标红就不看详情页；✗ 全文按版本逐节堆砌致顺序混乱；✗ 把各版本脚本混成无序列表。
- ✓ 路径摘要写明跨越了 `v6.3.0/v6.4.0/v6.5.0`；✓ 各版本后置脚本按序放进一次 config Pod 代码块；✓ `v7.1.0` 镜像命名变化合并到升级前。
